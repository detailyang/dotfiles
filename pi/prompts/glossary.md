# glossary

模糊说法 → 精确术语的映射表。消除 LLM 和人之间的术语歧义，确保理解错了会导致用错的词有明确对齐。

## 什么时候建

不是每个项目都需要。出现以下信号时建：

- 同一个词在代码里有多层含义（如 `cursor` = 字节偏移 vs 屏幕坐标）
- 同一概念有多个叫法（如 "缓冲区" vs "日志" vs "录制"）
- 用户说的词和代码里的类型名不对应
- 已经发生过因术语混淆导致的错误操作

没有这些信号就不建。

## 格式

文件路径：`docs/glossary.md`

```markdown
# Glossary

> 模糊说法 → 精确术语。与 LLM 对齐用。只记理解错了会导致用错的词。

## [分组名]

| 用法 | 精确术语 | 不是 |
|------|---------|------|
| 模糊说法 | **CodeLevelTerm** | 不是 X，不是 Y |
```

**三列规则**

1. **用法**：实际对话中出现的说法，多个用顿号分隔
2. **精确术语**：代码里的类型名、命令名、字段名，加粗，大小写与代码一致
3. **不是**：最关键的列。写排除项，不写概念解释

**收录标准**：只收理解错了会导致用错的词。不收行业通用词、上下文不会混淆的同名词、纯概念解释。

**分组**：按项目架构分层，不用固定模板（常见：架构、数据、操作）。组内按使用频率排序。

## 流程

**1. 读现有文件**：有 `docs/glossary.md` 就读取，了解已有术语避免冲突。如果项目有类型定义文件（`*.d.ts`、`schema.prisma`、`*.proto`）且容易获取，顺带扫一下——代码命名是判断歧义的基准。

**2. 识别歧义词**：每次只呈现一个候选，带推断让用户确认或纠正。有多个候选时逐个处理，不打包。

```
「Client」在代码里有两层含义：连接对象 vs 读取游标。你说的「Client」通常指哪个？
```

用户明确指定词时直接跳到步骤 3。

**3. 逐词确认**：每次只问一个。有候选项给选项，没有用反例逼近。拿到答案后复述确认。和行业惯例有出入时加备注，不否定用户选择。

**4. 写入**：按三列格式写入，追加到对应分组末尾。没有文件就新建并加文件头。

**5. 收尾**：

```
已更新 docs/glossary.md：新增 X、Y。
下次把这个文件丢给我，我会用里面的定义和你保持一致。
```

## 规则

- 每次只问一个问题，不打包多个
- 用户说"直接写"→ 跳过确认，写完附"请 review 以下定义"
- 新增歧义词追加条目，不改已有条目（除非用户明确要求）
- 无候选歧义词时，说明原因后停止，不继续轮询
